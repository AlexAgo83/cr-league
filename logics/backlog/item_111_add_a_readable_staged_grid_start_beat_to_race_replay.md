# Problem
- Playtest showed replay cars visually clustered under the start line at the beginning of a resolved Grand Prix.
- The replay is more time-real and cinematic after recent work, but the first seconds still read as a pile of cars instead of a race start.
- A simple deterministic start beat can improve clarity without changing race outcome or adding physics.

# Scope
- In:
  - Extend the existing replay plan/presentation layer in `ReplayView.tsx` to include a deterministic grid-start window before normal trace-driven movement.
  - Render starting cars with readable position offsets or staged progress offsets for the first seconds of replay.
  - Add a short launch/separation phase so cars transition from grid positions into existing replay movement smoothly.
  - Keep final classification, replay facts, tower order, events, and marker seeking deterministic and unchanged after the start window.
  - Update `ReplayView.test.ts` or related tests to cover start beat determinism and transition bounds.
  - Capture desktop/mobile replay-start screenshots or add e2e assertions for no car-label overlap at the start.
- Out:
  - Adding countdown audio or a 3D/canvas renderer.
  - Changing the race trace generated by simulation.
  - Changing qualifying grid calculation.
  - Making a full formation lap.

# Acceptance criteria
- AC1: At replay progress near zero, cars are separated enough to read their positions rather than fully stacked.
- AC2: The transition from staged grid positions into normal replay movement is deterministic and bounded.
- AC3: Seeking before, during, and after the start beat gives stable positions and tower state.
- AC4: Replay finish order and final classification remain unchanged.
- AC5: Tests or screenshots prove desktop and mobile replay starts are readable.
