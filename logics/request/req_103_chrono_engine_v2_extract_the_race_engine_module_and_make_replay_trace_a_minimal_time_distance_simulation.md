# Needs
- Extract the chrono race logic from simulateRace.ts into a focused internal module with a clear contract, exported pure helpers, and dedicated tests.
- Replace the reconstructed per-point replay movement with a minimal deterministic time-distance simulation trace that records each car's progress, speed, phase, elapsed time, order, and gaps from the same motion state.
- Make pit stops, overtakes, defense, risk, and event timing observable consequences of the simulated motion state where feasible, while preserving the arcade model and public RaceInput/RaceResult compatibility.
- Keep the implementation deliberately bounded: no full physics engine, no lane/collision model, no new runtime dependencies, no UI redesign, and no unrelated API/storage/release changes.
- Ship with strong invariant, determinism, replay, and balance evidence so another AI can implement the step safely from this corpus.

# Context
- The first chrono migration already moved final GP classification to deterministic chrono finish times. It added ChronoMotionParameters, CARD_MOTION_EFFECTS, motionParametersForDecision, chronoProfileFactor, chronoCircuitFactor, racecraftFactor, final-time coherence validation, and chrono balance metrics.
- The remaining architectural problem is that simulateRace.ts is now doing too much: segment score/event generation, chrono parameter mapping, finish-time integration, replay trace planning, replay smoothing, pit trace behavior, classification, and reporting all live in one large file.
- The remaining simulation fidelity problem is that replayTrace points are still generated by createDistanceReplayTrace from final times and TeamTracePlan. That is better than score-first ordering, but it is not yet a true captured trace of cars advancing over time/distance.
- Target shape: create an internal packages/shared/src/simulation/chronoRaceEngine.ts module. It should expose a small contract such as buildChronoMotionParameters, simulateChronoRace, and/or buildChronoReplayTrace. simulateRace remains the public entry point and should orchestrate existing narrative/events with the module output.
- A minimal time-distance simulation is enough. Use deterministic ticks or micro-segments over race distance. Each car state should include distanceMeters, speedFactor or speed, elapsedTime, phase, pit state, finished flag, and accumulated modifiers. Target speed comes from circuit speedProfile, weather, race traits, and motion parameters. Speed changes should be acceleration/deceleration bounded.
- ReplayTrace should be sampled from that state at the existing cadence or an intentionally documented cadence. For every sampled point, order should derive from distance/finish time, times should derive from elapsed race clock or finish crossing, gaps should derive from leader timing/distance, and cars[*] should reflect the sampled motion state.
- Pit behavior should become a state machine in the chrono module: pit_entry, pit_stop, pit_exit windows tied to normalized pitLaneProgress and pit strategy. Existing pit events can remain, but their traceProgress/trackProgress should align with the simulated pit state.
- Overtakes and defense should be minimal and deterministic. Only emit or annotate overtake phases when order actually changes between adjacent sampled states outside pit-related changes. Defense can remain a bounded racecraft modifier unless cars are close enough to mark defending; avoid continuous pack physics.
- Events should not be rewritten broadly. Existing weather/card/flavor event generation may remain in simulateRace, but event placement and replayFacts should consume chrono trace facts where available. Narrative events must not contradict phase, order, final times, or gaps.
- Balance risk from the first chrono pass remains visible: aggressive/high-upside options and mini_pack can dominate small samples, while defensive_order/economy/heavy_pack are conservative tradeoffs. The v2 implementation should preserve reporting and document whether tuning is required, but avoid broad economy redesign.

# Acceptance criteria
- AC1: Chrono race logic is extracted from simulateRace.ts into a focused internal module with documented inputs, outputs, tie-breakers, and migration boundaries; simulateRace remains the public entry point.
- AC2: ReplayTrace cars, order, times, gaps, phases, speeds, and final classification are sampled from one deterministic time-distance motion state rather than reconstructed only from final times.
- AC3: Pit phases, overtake phases, defense markers, finish events, replayFacts.orderChanges, and replay director beats align with actual trace state and actual order changes.
- AC4: Motion remains arcade-realistic and valid: finite times, no backwards progress, bounded speed and acceleration/deceleration, plausible pit entry/stop/exit windows, and deterministic output for repeated seeds.
- AC5: RaceInput/RaceResult remain compatible unless a minimal additive field is justified; API, storage, release automation, circuit catalogue, and UI layout remain untouched except for directly necessary replay helper adjustments.
- AC6: Dedicated tests cover the chrono module contract, parameter mapping, time-distance trace invariants, pit/overtake coherence, reward compatibility, replay helper compatibility, and deterministic multi-seed/circuit/weather/card/pit cases.
- AC7: Validation evidence includes typecheck, unit tests, build, lint, e2e, bounded balance simulation with chrono metrics, Logics validation, and a closeout report listing changed behavior and follow-up tuning risks.
